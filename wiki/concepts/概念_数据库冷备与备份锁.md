# 概念_数据库冷备与备份锁

## 定义

**冷备（Cold Backup）**：在数据库停止或部分停止写入的情况下，拷贝数据文件的备份方式。与热备（在线备份）和温备（只读备份）相对。

**备份锁（Backup Lock）**：MySQL 在执行冷备时施加的锁，用于防止备份期间数据文件被修改，确保备份数据的一致性。

## 冷备流程（MySQL + xtrabackup 为例）

```
1. 启动备份，施加 FLUSH TABLES WITH READ LOCK（备份锁）
2. 记录 binlog 位置点（show master status）
3. 拷贝数据文件（innobackupex 复制 .ibd、ibdata 等）
4. 记录 binlog position 和 innodb LSN
5. 释放锁，备份完成
6. 在备库应用 redo log，完成数据一致性恢复
```

> 注：如果是物理冷备（非 xtrabackup），流程类似但更依赖锁的持有时长

## 备份锁与 DDL 的冲突

- 备份锁本质上是**全局读锁**
- DDL（ALTER TABLE、CREATE INDEX 等）需要**写元数据锁（MDL）**
- 持有备份锁期间，所有 DDL 都会被**阻塞**

> 本次故障核心：备库冷备 → 持有备份锁 → 主库 DDL 过来 → 备库 SQL 回放被阻塞 → 半同步 ACK 无法返回 → 主库超时降级

## 大数据量冷备的挑战

| 问题 | 现象 | 缓解方案 |
|---|---|---|
| 备份时间长 | 锁持有时间过长 | 分库分表减少单库数据量 |
| 备份窗口与业务重叠 | 主从延迟、DML 堆积 | 调整到业务低峰期（如凌晨） |
| 底层 I/O 竞争 | 母机多实例争抢磁盘 | 错峰冷备、独立冷备母机 |
| 4T+ 规模 | 单次冷备超 1 小时 | 长期：拆库；短期：增量备份 |

## 备份锁与主从复制的交互

```
主库 ──(binlog)──→ 备库（冷备中，持备份锁）
                        ↓
                 收到 DDL event
                        ↓
                 尝试获取 MDL 写锁
                        ↓
                 被备份锁阻塞 → DDL 回放卡住
                        ↓
                 半同步 ACK 无法返回
                        ↓
              主库超时降级 → 主从延迟累积
```
