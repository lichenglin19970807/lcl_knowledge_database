# 概念_主从延迟

## 定义

主从延迟（Replication Lag）是指从库 Relay Log 回放（执行主库已提交事务）的速度，落后于主库提交新事务的速度。延迟越大，从库数据越"旧"，主从切换时潜在数据丢失越多。

## 量化指标

- `Seconds_Behind_Master`：从库 SQL 线程执行位置落后主库的秒数（最直观指标，但不准——备库 SQL 不跑时显示为 0）
- `Show Slave Status` 中的 `Slave_IO_Running` / `Slave_SQL_Running`：判断 IO 线程和 SQL 线程是否正常
- `pt-heartbeat`：用心跳表持续记录主库时间戳，精度更高

## 主从延迟的成因分类

### 1. SQL 回放阻塞型（本次故障类型）

备库 SQL 线程执行被阻塞（锁、备份、大查询），无法及时回放 binlog，导致延迟累积。

典型场景：
- 备库冷备持有备份锁（本次）
- 备库执行大事务（DDL、ALTER、批量 DELETE/UPDATE）
- 备库从机负载高，SQL 线程排队

**关键特征**：`Slave_IO_Running=Yes`，`Slave_SQL_Running=No`（或卡住）

### 2. 主库写入过载型

主库写入速度超过备库回放能力。常见于：
- 大批量 INSERT/DELETE
- 批量导入
- 业务高峰写入突增

**缓解**：备库升级配置、或加多从库分摊读压力

### 3. 网络带宽型

主从网络链路带宽不足或抖动，导致 binlog 传输延迟。

典型表现：`Slave_IO_Running` 变为 `Connecting` 或 `No`

### 4. 大事务型

单个事务过大（如单次 UPDATE 百万行），binlog 大且备库回放时间长。

## 主从延迟的风险

| 延迟程度 | 风险 | 可接受场景 |
|---|---|---|
| < 1s | 几乎无影响 | 读从库、报表 |
| 1s ~ 1min | 轻微数据不一致 | 低频分析 |
| > 1min | 切换可能丢数据 | 需警惕 |
| > 3000s | 严重告警，可能丢大量数据 | 紧急处理 |

## 监控与告警

- 日常：`Seconds_Behind_Master > 300s` 告警
- 重要业务：`pt-heartbeat` 延迟 > 60s 告警
- 本次故障：延迟超 3000s 触发告警
