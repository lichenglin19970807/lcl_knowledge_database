# 概念_读写竞态

## 定义

在分布式存储（如 HDFS / 对象存储）与计算框架（如 Spark / MapReduce）中，**读写竞态（Read-Write Race Condition）** 指的是当读取任务和写入任务在未加有效锁的同一时间窗口内操作同一批数据（如同一个 Hive 分区）时，导致读取失败或数据不一致的情况。

## Spark `INSERT OVERWRITE` 的 Commit 机制

造成大数据场景下竞态的一个核心微观机制在于 Spark 的 FileCommitProtocol：
1. **Staging（暂存）**：任务执行时，数据先写入以类似 `_temporary` 结尾的临时目录。
2. **Commit（提交）**：Task 成功后，将临时文件 Rename 到正式路径。
3. **Clean（清理）**：如果是 OVERWRITE 语义，还会清理掉之前存在的旧版文件。

## 经典报错：FileNotFoundException

当上游在执行 Commit/Clean 动作时，如果下游刚好启动，极易发生 `FileNotFoundException`。

**典型时序窗口重叠**：
1. **下游 Split 阶段**：MR 或 Spark 任务首先进行 Split，也就是遍历 HDFS 目录拿到文件列表（如拿到 uuid 格式的文件名）。
2. **另一上游 Commit 阶段**：此时另一个并发写入任务刚好执行 Commit，删除了上一步中的 uuid 旧文件，并 Rename 出了新文件。
3. **下游 Map 阶段**：Map 任务尝试依据最初拿到的文件清单打开文件，发现文件已经被清理，抛出 `FileNotFoundException`。

## 解决与防范

1. **宏观层面（推荐）**：通过调度依赖系统强制串行，或遵循 [[概念_SingleWriter原则]] 避免并发写入。
2. **任务层面**：下游在失败后配置自动重试机制（通常等上游 Commit 完全结束后重试可“自愈”）。
3. **存储引擎**：使用支持 ACID 事务的现代数据湖架构（如 Apache Iceberg, Hudi），它们通过快照隔离级别（Snapshot Isolation）从根本上解决了底层文件的读写竞态。
