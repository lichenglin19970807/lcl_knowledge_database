# Hive表被两条调度链并发写入导致读写竞态

## 来源信息

- **原始文件**：`raw/Hive 表被两条调度链并发写入导致读写竞态.md`
- **日期**：2026-04-24

---

## 故障现象

- **链路**：周计算 (PythonSQL) → 滚动分区 → Hive 出库 MySQL
- **报错**：出库任务在 Map 阶段报 `FileNotFoundException`，18分钟后自动重试成功。
- **线索**：失败文件名带有 UUID 和特定行数特征，且该行数等同于最终成功重试时的行数（说明读到的并非“上游写了一半的半成品”，而是最终版）。

## 根因排查与时序分析

最初怀疑调度尾部窗口等小概率事件，但最终通过**全平台任务反查**发现，该表存在**两条并行的独立调度链**（天任务、周任务），均使用了同一份 PythonSQL 脚本对同一个 Hive 分区进行 `INSERT OVERWRITE` 操作。

**冲突时序（周日凌晨）：**
1. **周计算** 和 **天计算** 同时开跑。
2. **周计算** 率先完成（02:01:29）。
3. **周出库** 被触发启动，并进入 MR split 阶段（生成需要读取的文件清单）。
4. **天计算** 稍晚完成（02:01:57），触发 Commit 阶段操作：rename 临时文件到正式目录，并**清理旧文件**。
5. **周出库** 的 Map 尝试打开清单中的 uuid 文件，但该文件已被天计算删掉/覆盖，导致 `FileNotFoundException`。

## 架构层面的缺陷

该故障是两个独立架构缺陷叠加的产物：

1. **隐形并发写入**：同一张表有天、周两条无依赖关系的调度链路并发写入，全靠“碰巧错开”。
2. **DAG 失联（调度盲区）**：US（统一调度）的依赖图只记录了配置好的依赖。调度器认为上游 SUCCESS 就安全了，完全不知道有另一条链在“侧面杀入”。调度系统的世界观里不存在这个 Bug。
