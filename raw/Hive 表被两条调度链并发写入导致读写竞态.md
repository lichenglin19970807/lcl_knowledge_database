# Hive 表被两条调度链并发写入，导致读写竞态
## 一、现象

一条典型的 TDW 周度出库任务链：

```
周计算 (PythonSQL, 写 Hive 表)
    ↓
Hive 滚动分区 (维护分区保留策略)
    ↓
Hive 出库 MySQL (TDW→MySQL)
```

- 出库任务 `20240906155837406`，周调度，数据日期 `2026-04-13`，实际执行日 `2026-04-20 凌晨`
- **首次执行失败**：YARN Map Attempt 均报 `FileNotFoundException`，Hive ExecDriver return code 2
- 18 分钟后自动重试成功
- 失败文件名特征：`part-00000-fe07627e-...-59818.orcf`（文件名里的 uuid 和行数是关键线索）

表面上看：父任务 US state=SUCCESS，子任务紧接着启动却读不到文件，然后自愈。看起来像调度小概率抖动，但发生频率 25% 显然不对劲。

---

## 二、排查历程

### 第 1 轮：怀疑"上游还在写"（读写竞态经典解释）

最初的直觉归因：源表分区正在被上游 ETL 覆盖写，出库读到了半成品文件。

**被推翻**：失败文件名 `part-00000-fe07627e-...-59818.orcf` 中的 `-59818` 正好等于最终成功重试时读到的行数。这说明出库读到的**就是最终版文件**，不是半成品。"上游还在写"这个解释不成立。

### 第 2 轮：怀疑"Spark commit protocol 的尾窗口"

退一步解释：US 层的 "SUCCESS" ≠ HDFS 文件视图对下游完全稳定。Spark FileCommitProtocol v1 存在 rename staging → 正式路径的短暂窗口，下游如果紧贴着读可能撞进去。

**用户关键质疑**："一个道理啊，滚动分区任务完成了，才会执行出库任务啊。父任务 02:01:29 已经成功了，US 调度就是这样设计的。"

这一质疑逼出了关键反思：虽然 Spark commit protocol 确实存在理论上的尾窗口，但这通常是**亚秒级**，解释不了 "02:01:29 SUCCESS → 02:02:14 文件消失" 这个 45 秒的空档。滚动分区任务按配置只动"保留期外的老分区"，不该动本周新分区。这一版解释**证据不硬**。

### 第 3 轮：跳出 US 依赖图，扫全平台

真正的突破发生在用户主动追问："**还有哪个任务会动这张表？**" —— 这句话直接改变了排查方向，从"解释现象"转向"找真正动手的人"。

关键动作：不再盯着声明的父子依赖，而是用 task-list 跳出原依赖图扫全平台：

```bash
do-bigdata us task-list --task-name ads_tm_enterprise_saas_resource_dept_stats_wi --page-size 50
```

结果震惊：**一共 11 个任务跟这张表相关**，其中一条完整的**天调度链**与周链完全独立运行，而且——

| 任务 ID | 任务名 | 类型 | 周期 | 脚本名 |
|---|---|:-:|:-:|---|
| `20240906113354887` | ads_tm_...dept_stats_wi | PythonSQL | **周** | `ads_tm_enterprise_saas_resource_dept_stats_wi.py` |
| `20240906163613801` | ads_tm_...dept_stats_wi **日执行** | PythonSQL | **天** | `ads_tm_enterprise_saas_resource_dept_stats_wi.py` ⚠️ **同一份脚本** |

天任务和周任务：**同一份 .py 脚本、相同参数、写同一张表**，只是调度周期不同，完全并行运行。

## 三、问题结论

### 两条调度链的全貌

**周调度链**（出库任务感知到的）：
```
周计算 20240906113354887  (PythonSQL, 写 Hive 表, 周)
    ↓
滚动分区 20240906155645985  (周)
    ↓
出库 20240906155837406  (TDW→MySQL, 周)
```

**天调度链**（出库任务**完全不感知**）：
```
天计算 20240906163613801  (PythonSQL, 同一份脚本, 写同一张表, 天)
    ↓
into_table_data_to_mysql 日执行 20240906163925108  (天)
    ↓
into_ads_tm_... 20240906164016301  (天)
```

### 冲突发生在哪里

**两条链每周日凌晨必然同时执行**（周任务每周日跑 + 天任务每天跑），且都往同一张表的分区写数据：

1. **同时启动**：周日凌晨 01:59:30，周计算和天计算同时开跑（它们祖父任务相同）
2. **commit 错峰**：周计算 02:01:29 先完成，天计算 02:01:57 后完成（差 28 秒）
3. **出库抢跑**：周链在周计算完成 9 秒后就触发了出库任务（02:01:38）
4. **撞窗口**：出库的 MR split 阶段先于天计算的 commit，但 Map Attempt 打开文件晚于天计算的 commit
5. **天计算 commit 动了 split 清单里的文件**：Spark INSERT OVERWRITE 在 commit 阶段会 rename 临时文件到正式名、清理上一版旧文件。出库 split 清单里持有的 `fe07627e` 这个 uuid 文件，被天计算的 commit 替换/清除，Map 打开时就找不到了

### 为什么 US 依赖机制防不住

- 出库任务的 US 父依赖只配了滚动分区任务，**天任务链根本不在它的依赖图里**
- US 调度系统**不知道**还有另一条链也在写同一张表
- 即便周链严格按"父成功→子启动"执行，天链从侧面杀进来，US 调度层完全无感
- 这不是调度 bug，是依赖声明不完整导致的 DAG 失联

## 四、总结与思考

### 架构层面：隐形并发写入 + 调度层 DAG 失联

这个 case 的本质不是调度抖动、不是 HDFS 延迟、不是 Spark commit 尾窗口，而是**两个独立架构缺陷的叠加**：

1. **隐形并发写入**：同一张 Hive 表有两条独立调度链（天 + 周）用同一份脚本写入，且没有任何同步/互斥机制。两条链能否不冲突完全靠"碰巧执行时间错开"
2. **调度层 DAG 失联**：US 依赖图只反映"被声明"的依赖。出库任务的 owner 配依赖时只想到周链内部的父子关系，完全不知道侧面还有一条天链也会写这张表。依赖图 ≠ 真实依赖

两个缺陷叠加的结果：**上游所有任务 US 状态都是 SUCCESS，下游依然必然间歇性失败**，而且从 US 依赖图里看不出任何异常。这是最难排查的一类故障——**调度系统的世界观里根本不存在这个 bug**。

### 设计思考：Single Writer Principle

> **"一张表应该有唯一的真实写入者。违反这个原则时，任何下游读者都无法安全读取。"**

这是数据仓库设计中一条朴素但容易被违背的原则。它的含义是：

- **一张表 = 一个 owner = 一条写入链路**。所有对这张表的写入必须通过这一条链路，其他需求（日刷新 / 周刷新 / 补数 / 修数）也必须纳入这条链路的调度范畴
- **下游读者无需担心谁在写**：只要表存在、分区就绪，读取就是安全的
- **并发写入必须显式建模**：如果业务上真的需要多个任务写同一张表（比如分区互不重叠的多源合并），必须在调度层显式声明彼此的互斥或顺序关系，让 DAG 知道它们的关系

违反 Single Writer 原则的代价不是"偶尔失败"，而是**下游的任何读操作都变成不确定的**。
