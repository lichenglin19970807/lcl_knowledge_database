# MySQL 主备架构

- **来源：** 案例「仪表盘数据库冷备导致主从延迟」/ 20260324
- **类型：** 数据库 / 知识点

---

## 常见架构模式

### 1主1备2只读

```
主库 (Master)
  └── 备库 (Slave/Standby)  ← 用于容灾切换
  └── 只读实例1 (Read Replica)  ← 业务读流量
  └── 只读实例2 (Read Replica)  ← 业务读流量
```

- 业务写：走主库
- 业务读：走只读实例，分担主库压力
- 容灾：主库故障时切换到备库

### 1主2备（更高可用）

相比 1主1备，多一个备库，冷备/备份操作不影响唯一备库的可用性，提升容灾能力，但成本更高。

---

## 主从同步机制

### 半同步复制（Semi-sync Replication）

- 主库写入后，**等待至少一个备库 ACK** 再返回成功
- 若等待超时，**自动降级为异步复制**，继续提交
- 保障：在超时降级前，至少有一个备库收到数据，减少数据丢失风险

### 异步复制（Async Replication）

- 主库写入后立即返回，不等备库 ACK
- 性能最好，但主库故障时可能丢数据

---

## 冷备对主从同步的影响

冷备期间，备库持有**备份锁（Backup Lock）**：

```
冷备开始
  → 备库持有备份锁
  → 主库执行 DDL
  → 主库向备库同步 DDL
  → 备库 DDL 被备份锁阻塞，无法回放
  → 半同步等待 ACK 超时
  → 降级为异步，主从延迟开始积累
冷备结束
  → 备份锁释放
  → 备库追平延迟
```

**注意：** 冷备期间如主库发生故障，此时主从切换会有丢数据风险。

---

## 最佳实践

1. **冷备时间点选业务低峰**（如凌晨）
2. **监控主从延迟**，设置合理告警阈值，并在冷备窗口期设置告警屏蔽
3. **数据规模大时考虑拆库**，减少单次冷备时长
4. **高可用要求高时，用 1主2备**，保证冷备不影响容灾能力
