# 引用分离

- **来源：** 案例「web端修改会议邀请人优化」/ 20260324
- **类型：** 服务治理 / 知识点

---

## 定义

将**体积大、不频繁访问、或非核心业务逻辑依赖的数据**，从主协议/消息/数据库表中剥离，用一个可寻址的**引用（ID/Key/URL）**来替代，数据本身单独存储在独立服务/存储系统中。

---

## 核心思想

```
原来：主链路携带完整数据 → 数据量大时造成限制或性能问题

优化：主链路只携带引用 ID → 需要时再通过 ID 取回完整数据
```

---

## 适用场景

| 场景 | 说明 |
|------|------|
| 协议包体限制 | 如 UDP 包大小限制、消息队列单条大小限制 |
| 数据库大字段 | 将 BLOB/TEXT 大字段存到对象存储，DB 只存 URL/Key |
| 消息队列 | MQ 消息体不宜过大，大数据用引用传递 |
| HTTP 请求体 | 上传大文件时先上传到 OSS，接口只传文件 ID |
| 微服务调用 | 跨服务传递大对象时，用引用代替序列化传输 |

---

## 优点

1. **解耦数据大小与主链路**：主链路轻量，不受包体/消息体大小限制
2. **按需加载**：不是每次都需要完整数据，减少不必要的传输
3. **复用性**：同一份数据可被多个下游引用，避免重复传输
4. **存储灵活**：大数据可选择更适合的存储介质（对象存储、专用服务等）

## 缺点

1. **额外一跳**：需要时要额外请求数据，增加延迟
2. **一致性管理**：引用的数据需要管理生命周期（过期清理、版本一致性）
3. **复杂度增加**：需要额外的存储服务和 ID 管理机制

---

## 实际案例

- 会议邀请人列表：从 UDP RPC 包中剥离，存入邀请人服务，主链路传临时 ID
- 图片/视频上传：先传 OSS，业务接口只传文件 URL
- 大 JSON 配置：存配置中心，服务启动时通过 Key 拉取
